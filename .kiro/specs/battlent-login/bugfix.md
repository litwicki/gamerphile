# Bugfix Requirements Document

## Introduction

The Battle.net OAuth login flow is broken. When a user attempts to sign in via Battle.net, the authorization URL generated by next-auth is missing the required `state` query parameter. Battle.net's OAuth server rejects the request with an HTTP 400 error: "The state parameter must be provided."

The root cause is in the custom Battle.net provider (`lib/auth/battlenet-provider.ts`), which is configured with `type: "oidc"`. In next-auth v5 beta 30, OIDC providers rely on OpenID Connect discovery from the issuer (`https://oauth.battle.net`). However, Battle.net's OIDC discovery endpoint or the interaction between the OIDC flow and the custom `authorization.params` configuration is causing the `state` parameter to be omitted from the generated authorization URL. Switching the provider type from `"oidc"` to `"oauth"` and explicitly specifying the authorization and token endpoints (along with the required `state` parameter handling) should resolve the issue.

## Bug Analysis

### Current Behavior (Defect)

1.1 WHEN a user initiates Battle.net OAuth login THEN the system generates an authorization URL missing the required `state` query parameter, resulting in an HTTP 400 error from Battle.net ("The state parameter must be provided")

1.2 WHEN the authorization URL is constructed using the `type: "oidc"` provider configuration with issuer `https://oauth.battle.net` THEN the system fails to include the `state` parameter in the authorization redirect URL

### Expected Behavior (Correct)

2.1 WHEN a user initiates Battle.net OAuth login THEN the system SHALL generate an authorization URL that includes a valid `state` query parameter, allowing Battle.net to accept the request and proceed with the OAuth flow

2.2 WHEN the authorization URL is constructed for the Battle.net provider THEN the system SHALL include all required OAuth parameters (`response_type`, `client_id`, `redirect_uri`, `scope`, `state`) in the authorization redirect URL

### Unchanged Behavior (Regression Prevention)

3.1 WHEN a user completes the Battle.net OAuth flow successfully THEN the system SHALL CONTINUE TO store the access token in the JWT and make it available in the session via the `jwt` and `session` callbacks

3.2 WHEN a user authenticates via Battle.net THEN the system SHALL CONTINUE TO extract the `battletag` from the profile and include it in the session

3.3 WHEN the OAuth callback is received at `/api/auth/callback/battlenet` THEN the system SHALL CONTINUE TO process the callback and establish a valid user session

3.4 WHEN the authorization request is made THEN the system SHALL CONTINUE TO request the `openid wow.profile` scopes
